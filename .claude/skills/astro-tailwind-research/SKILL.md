## 目的

「なんとなく知っている書き方」で実装して後から手戻りするのを避けるため、実装に着手する前に**インストール済みのバージョンに合った**Astro/Tailwindの正しい書き方を調べ、設計メモとして残す。特にTailwind CSS v4はv3からConfigの書き方（CSSベースの`@theme`など）が大きく変わっており、Astro 5のContent Collectionsも旧バージョンとAPIが異なるため、バージョンを確認せずに調べると古い情報に引きずられやすい。

## 手順

1. **対象機能を特定する**: 何を実装したいか（例:「3カラムレイアウトをレスポンシブにしたい」「GitHub contributionグラフ風のヒートマップを作りたい」）を明確にする。`TODO.md` に該当セクションがあれば読む。
2. **現在のバージョンを確認する**: `package.json` を読み、`astro` / `tailwindcss` / `@tailwindcss/vite` / `@astrojs/mdx` 等の実際のバージョンを確認する。検索・参照するドキュメントはこのバージョンに合わせる（例: Tailwind v4なら`@theme`や`@import "tailwindcss"`ベースの書き方、v3の`tailwind.config.js`ベースの情報は混同しない）。
3. **公式ドキュメントを中心に調べる**: `WebSearch`/`WebFetch` で以下を優先する。
   - Astro公式ドキュメント（docs.astro.build）
   - Tailwind CSS公式ドキュメント（tailwindcss.com/docs）
   - 該当ライブラリの公式README/ドキュメント（例: `@tailwindcss/typography`, `astro-expressive-code`, `p5.js`）
   - 上記で解決しない実装の勘所（レスポンシブの崩れ方、既知の落とし穴など）は技術ブログ等の二次情報も参照してよいが、コードは公式ドキュメントの記法を優先する。
4. **設計メモを書く**: `docs/design-notes/<topic-slug>.md` に、下記のレポート構造で保存する。ファイルが既にあれば追記・更新する。
5. **元のTODO.mdとの対応を明記する**: `TODO.md`の該当セクション番号を「関連TODO」として記載し、実装時に相互参照できるようにする。

## レポート構造

必ず以下のテンプレートで書く。

```markdown
# <機能名>の実装方針

関連TODO: TODO.md §<番号>

## 課題
（実現したいこと、現状の制約）

## 推奨アプローチ
（採用する実装方針とその理由。複数の選択肢があった場合は簡潔に比較する）

## 実装スケッチ
（実際に使うコード例。ファイルパスやコンポーネント名も具体的に書く）

## 注意点・落とし穴
（ハマりやすい点、パフォーマンス、既存コードとの衝突など）

## 参考リンク
（参照した公式ドキュメント等のURL）
```

## 心構え

- 動くコードを書くことが目的ではなく、**後で実装する自分（またはClaude）が迷わないための地図を作る**ことが目的。実装スケッチは最小限の骨組みでよく、細部まで完成させる必要はない。
- 複数の実装候補がある場合、無理に1つに絞らず「Aがシンプルだが将来的な拡張にはBが向く」のように選択の余地を残してよい。最終判断はユーザーと相談する。
- バージョン固有の情報であることを意識し、「Tailwind v4では」のように明記する。将来バージョンアップした際に「これはv4時点の情報」と分かるようにしておく。
